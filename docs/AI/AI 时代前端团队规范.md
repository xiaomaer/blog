# AI 时代前端团队规范（建议版最佳实践）

目标不是“限制大家”，而是：

## 保证：

**AI 提效 + 工程质量 + 可维护性 + 可持续交付**

避免：

> AI 写得越快
> 项目死得越快

---

# 一、AI 使用规范（最高优先级）

---

## 1. Prompt 模板规范（必须统一）

所有需求都必须使用统一模板：

【业务背景】

【业务目标】

【用户场景】

【输入数据】

【输出结果】

【边界条件】

【异常情况】

【技术栈要求】

【代码风格要求】

【禁止事项】

【验收标准】

---

## 最佳实践

禁止：

> “帮我写个页面”

必须：

> 可描述
> 可验证
> 可验收

否则：

AI 输出不可控。

---

## 2. AI 代码使用分级制度（必须）

建议分为：

| 等级 | 场景                | 是否允许 AI 直接提交 |
| -- | ----------------- | ------------ |
| L1 | 页面骨架 / CRUD / 表单  | 可以           |
| L2 | 普通业务逻辑            | 必须 Review    |
| L3 | 权限 / 支付 / 风控      | 禁止直接提交       |
| L4 | 基础架构 / SDK / 核心链路 | 必须人工主导       |

---

## 最佳实践

规定：

### AI 生成代码必须标注来源

例如：

```ts
// Generated with AI, reviewed by Ethan, 2026-04-24
```

方便追踪责任。

---

# 二、代码规范（核心）

---

## 3. Git 提交规范（必须）

统一 Commit Message：

feat:
fix:
refactor:
perf:
test:
docs:
style:
chore:

例如：

feat: 新增订单列表筛选功能
fix: 修复登录状态失效问题
refactor: 重构用户权限模块

---

## 最佳实践

禁止：

```text
update
修改一下
fix bug
重新提交
```

这种提交没有任何价值。

---

## 4. 分支管理规范

推荐：

main
develop
feature/*
release/*
hotfix/*

最佳实践：

* main 永远可发布
* develop 用于集成测试
* feature 独立开发
* hotfix 快速线上修复

---

## 5. 目录结构规范（必须）

建议：

```text
src/
 ├── api/
 ├── components/
 ├── hooks/
 ├── pages/
 ├── layouts/
 ├── store/
 ├── utils/
 ├── constants/
 ├── types/
 ├── services/
 └── tests/
```

---

## 最佳实践

禁止：

```text
utils2/
newComponents/
test-final/
aaa/
temp/
```

这种目录会毁掉项目。

---

# 三、组件规范（重点）

---

## 6. 组件设计规范

必须统一：

### 单一职责原则

一个组件：

只做一件事。

禁止：

```tsx
UserPage
```

3000 行大组件。

必须：

```tsx
UserInfoCard
UserPermissionPanel
UserOrderTable
```

拆分明确。

---

## 7. Props 规范

必须：

### 类型明确 + 默认值明确 + 禁止透传污染

例如：

```ts
interface Props {
  visible: boolean
  onClose: () => void
}
```

禁止：

```ts
props: any
```

这是灾难。

---

## 8. 状态管理规范

原则：

## 能局部状态，就不要全局状态

优先级：

```text
组件状态
↓
页面状态
↓
模块状态
↓
全局状态
```

不是：

一上来就 Redux。

---

# 四、Code Review 规范（决定上限）

---

## 9. Review Checklist（必须）

Review 不看：

“能不能跑”

重点看：

### 必查项

* 类型安全
* 边界处理
* 空状态
* Loading 状态
* 错误处理
* 性能问题
* 重复渲染
* 可测试性
* 可维护性
* 安全风险
* 可访问性（a11y）

---

## 最佳实践

要求：

### AI 先审 + 人再审

双重保障。

---

# 五、测试规范（不能省）

---

## 10. 测试策略

必须：

### 核心链路必须覆盖

建议：

| 类型          | 覆盖内容         |
| ----------- | ------------ |
| Unit Test   | 工具函数 / Hooks |
| Integration | 页面逻辑         |
| E2E         | 登录 / 支付 / 下单 |

---

## 最佳实践

禁止：

> “这个功能很简单，不用测”

线上事故都这么来的。

---

# 六、上线规范（非常重要）

---

## 11. 发布 Checklist（必须）

上线前必须确认：

构建通过
单测通过
E2E 通过
埋点完成
错误边界处理完成
监控告警已配置
灰度方案明确
回滚方案明确
文档已更新

---

## 最佳实践

禁止：

> “先上线再说”

这是事故制造机。

---

# 七、文档规范（长期价值）

---

## 12. 文档必须标准化

必须有：

* 需求文档
* 技术方案文档
* 接口文档
* 发布文档
* 回滚方案
* 故障复盘文档

---

## 最佳实践

文档不是给领导看的：

## 是给未来的自己看的

尤其：

半年后的你。

---

# 最后一条（最重要）

## 规范不是为了限制人

而是为了：

# 让优秀的人持续高效地产出

没有规范：

优秀的人会被低质量协作拖死

有规范：

团队才真正具备：

## 可复制的交付能力
