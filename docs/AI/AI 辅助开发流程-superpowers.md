# AI 辅助开发流程：Superpowers（以 Cursor 为例）

> 上游项目：[obra/superpowers](https://github.com/obra/superpowers)  
> 本文基于该仓库公开说明整理，并结合 **Cursor** 中的安装与使用方式撰写。

---

## 1. 什么是 Superpowers

**Superpowers** 是一套面向「编程智能体」的软件开发方法论：在可组合的 **Skills（技能）** 之上，配合一组初始指令，让智能体在会话中 **自动识别并遵循** 这些技能，而不是凭默认习惯直接写代码。

其核心体验可以概括为：

- 一启动对话、一旦进入「要做功能/改行为」的场景，智能体 **先退一步**，通过对话把你要做的事澄清成可执行的规格（spec）。
- 规格经你确认后，再产出 **足够细、足够可执行** 的实现计划（面向「缺少项目上下文、测试习惯一般」的执行者也能照做）。
- 执行阶段强调 **真红/绿 TDD**、**YAGNI**、**DRY**；并可配合 **子代理驱动开发** 等流程，让智能体在既定计划下较长时间自主推进。

技能会在相关场景下被触发，你不必每次手动背诵整套流程——安装并启用插件后，**Cursor 里的 Agent 会按技能库约定行事**。

---

## 2. 为什么需要 Superpowers

常见痛点是：智能体「接话就写」，容易在需求未对齐、设计未确认时大量产出代码，后续返工成本高。

Superpowers 试图系统性解决几类问题：

| 问题 | Superpowers 的应对思路 |
|------|------------------------|
| 需求含糊就开工 | **brainstorming** 等流程：先对话澄清，再呈现设计并要用户点头 |
| 计划过大、难以落地 | **writing-plans**：拆成 2–5 分钟粒度的小步，含路径、验证方式 |
| 实现偏离规格 | **subagent-driven-development** 等：分任务 + 两阶段评审（规格符合性 → 代码质量） |
| 「修好了」但未验证 | **verification-before-completion**：完成声明前要有证据 |
| 调试凭感觉 | **systematic-debugging**：分阶段根因分析，而非乱试 |

官方 README 中的理念概括包括：**测试驱动**、**系统化优于临场发挥**、**以简化为目标**、**证据优于口头宣称**。这些通过技能文件落地为 **强制工作流，而非建议**。

---

## 3. 适用场景

更适合：

- **新功能、组件、行为变更**：需要先从想法收敛到设计与规格。
- **多步骤、多文件改动**：需要可检查、可追踪的实现计划。
- **希望强调 TDD 与可验证交付** 的团队或个人习惯。
- **希望在 Cursor 里用插件统一「做事方式」**，减少每个项目单独写长篇 `AGENTS.md` 却仍被模型跳过的情况。

相对不那么「必须」的场景：

- 纯信息问答、单文件极小改动且你已明确每一步（仍可受益，但收益相对小）。
- 完全无法接受「先设计/先计划再写代码」节奏时，会与部分技能的硬门槛冲突。

## 4. 准备工作

### 4.1 环境

- 已安装 **Cursor**，并能使用 **Agent 对话**（插件与 Agent 能力以 Cursor 当前版本为准）。
- 本地仓库建议具备：可运行的测试命令（若你希望完整走 TDD 与验证类技能）、清晰的目录结构（便于计划中写清路径）。

### 4.2 与项目规则的优先级

上游 **using-superpowers** 技能说明：**用户显式指令**（如 `AGENTS.md`、`CLAUDE.md`、对话里的直接要求）优先于技能；技能仅在不与用户冲突时覆盖默认系统行为。  
在 Cursor 中，你仍可通过 **项目规则 / 用户规则** 声明例外（例如某项目明确不写测试）。

### 4.3 默认产物路径（来自上游技能约定）

部分技能会引导将文档写到仓库内固定位置（可按团队规范改写路径，但需与 Agent 说明一致）：

- 设计/规格：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 实现计划：`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

若你的博客或单体项目没有该目录，可在首次使用前创建，或在规则里指定替代路径。

### 4.4 技能可以选用吗

1. 你主动选  
在聊天里用 / 调出对应技能（例如 TDD、brainstorming、writing-plans 等），等于指定「这次按这条技能走」。

2. Agent 自动选  
插件说明里也写了：和任务相关时，Agent 可以自动套用合适技能，不一定要你每次都点。

3. 你说了算  
using-superpowers 里写得很清楚：用户明确写的规则（如 AGENTS.md、Cursor 规则）优先于 Superpowers。
所以你可以在自己的规则里写「默认不用 TDD」「某类任务不用 brainstorming」等，和 Superpowers 不冲突时以你为准。

4. 技能里的「例外」  
例如 TDD 技能允许在征得你同意的前提下对纯配置、一次性原型等豁免——这也是「可选用」的一种体现。

**简要结论：** Superpowers 技能可以选用（/技能名 或你写在项目规则里）；不是每条对话都必须开满所有技能，你的指令和项目规则优先。

### 4.5 使用说明（建议做法）

1. **先决定范围**：全仓库默认跟 Superpowers，还是仅某类分支/项目启用；与团队对齐是否强制 TDD、规格与计划目录。  
2. **用规则写清例外**：在 Cursor **项目规则**或 `AGENTS.md` 中写：哪些情况可以缩短设计、哪些目录不写 `docs/superpowers/`、测试命令是什么——减少每轮对话重复说明。  
3. **会话内显式收口**：若某次任务确定不走完整流程，在 **首条消息或同一条需求里** 写清楚（例如「仅格式化，不要生成设计文档」），避免与 Agent 的「硬门槛」预期打架。  
4. **路径与产物一致**：若改了默认 `docs/superpowers/...`，在规则里写新路径，避免 `writing-plans` / `brainstorming` 落盘位置与团队习惯不一致。  
5. **能力对齐**：子代理、并行任务、跑测试等能力依赖 Cursor 与本地环境；上游也提示无子代理时执行质量可能下降——必要时拆会话或人工承担「检查点」角色。  
6. **升级与差异**：插件与 Cursor 版本更新后，若行为与文档不符，以 **Cursor 与插件市场说明** 为准；技能正文以仓库内 `SKILL.md` 为准。

### 4.6 示例：规则与对话中的例外

以下为 **示意**，复制到仓库后请按你们真实目录、分支策略和测试要求改写。Cursor 中同等内容也可写在 **项目规则（Project Rules）** 里，与 `AGENTS.md` 二选一或并存（以你们 Cursor 版本如何加载为准）。

#### 示例 A：仓库级 `AGENTS.md`（路径 + 可跳过 brainstorming 的条件）

```markdown
## 与 Superpowers 共存的仓库约定

- 本仓库使用 Superpowers；**用户在当前对话中的明确指令**优先于本节。
- 设计/规格落盘目录：`docs/internal/designs/`（不使用默认 `docs/superpowers/specs/`）。
- 实现计划落盘目录：`docs/internal/plans/`（不使用默认 `docs/superpowers/plans/`）。
- 满足以下全部条件时，可 **跳过 brainstorming 的落盘与多轮方案对比**，仅在对话中用一两句话确认后直接改代码：
  - 仅单文件修改，且不改变对外 API 或用户可见行为；或
  - 仅限 typo、格式化、注释调整。
- 验证：合并前至少执行 `pnpm test`（若任务声明免测，须由需求提出方在对话中写明原因）。
```

#### 示例 B：同文件内声明「不写测试」的例外（需与人对齐）

```markdown
## 测试

- 默认遵循 Superpowers 的 TDD 节奏；以下情况可在任务中显式声明 **本任务不写新测试**：
  - 纯文案、纯配置、或依赖外部沙箱无法单测的脚本；
  - 须在同一对话中由需求方确认「免测」理由。
```

#### 示例 C：单次对话首条消息（跳过部分流程）

适合「你心里已完全定案、只想 Agent 执行」时使用；写得越具体，与技能默认流程冲突越小。

```text
任务：在 README.md 末尾增加「常见问题」一节，共 3 条 Q&A。

约束：跳过 brainstorming 与独立设计文档；不需要 writing-plans；直接修改 README.md。
改完后运行 `pnpm exec prettier --write README.md` 即可，无需跑全量测试。
```

#### 示例 D：仅收窄「完成」声明（配合 verification-before-completion）

```text
在声称「测试全部通过」之前，必须在本对话中贴出你刚执行的完整命令行和退出码；
本项目固定使用：`pnpm test`（不要用 npm test）。
```

## 5. 快速开始（Cursor）

根据 [GitHub 仓库 README](https://github.com/obra/superpowers) 中 **Cursor (via Plugin Marketplace)** 一节：

1. 打开 **Cursor Agent** 对话。
2. 通过插件市场安装，任选其一：
   - 在对话中使用：`/add-plugin superpowers`
   - 或在插件市场中搜索 **「superpowers」** 并按提示安装。

安装完成后，**无需额外「开关」**：当对话内容触发某技能的描述场景时，Agent 应按该技能工作（具体行为取决于 Cursor 对插件技能的加载方式；若某版本有差异，以 Cursor 官方说明为准）。

**首次验证建议**：提一个「小功能但需多文件」的需求，观察 Agent 是否先进入澄清/设计，而不是直接改代码。

技能是否必选、如何用规则收窄或声明例外及 **示例**，见 **§4.4～4.6**。

## 6. 完整工作流程

以下与官方 README **The Basic Workflow** 及技能库设计一致，并标注在 Cursor 中的典型对应关系。

### 6.1 头脑风暴与设计（brainstorming）

- **时机**：任何「创造性工作」之前——新功能、新组件、改行为等。
- **要点**：在呈现设计并得到你批准之前，**不**应进入实现类技能或写实现代码；可给出 2–3 种方案与取舍；设计分段展示便于阅读；最终落盘规格并请你再读一遍规格文件。
- **出口**：转入 **writing-plans**（不要在 brainstorming 结束后直接跳到某个具体实现技能）。

### 6.2 隔离工作区（using-git-worktrees，可选但推荐）

- **时机**：设计获批之后。
- **作用**：在新分支/独立 worktree 中工作，跑一遍项目初始化与「干净基线下测试」，减少与主分支并行修改的干扰。

### 6.3 编写实现计划（writing-plans）

- **时机**：已有经批准的规格/需求，且准备动代码之前。
- **要点**：假设执行者对代码库与工具链几乎不熟；任务粒度约 **2–5 分钟一步**；每步包含文件路径、如何验证；贯彻 DRY、YAGNI、TDD；计划保存到约定路径（见上文）。

### 6.4 执行计划：两种主路径

- **subagent-driven-development**：适合「计划已有、任务相对独立、希望在本会话内由子代理逐条执行」；每条任务后 **两阶段评审**（先规格符合性，再代码质量）。
- **executing-plans**：适合「并行会话、分批次执行、带人类检查点」的节奏。

在 Cursor 中，若界面支持 **Task / 子代理**，则与 upstream 描述的「每任务干净上下文」更一致；若当前版本以单会话为主，仍可手动按计划中 checkbox 逐步让 Agent 执行，并自行在关键节点做评审。

### 6.5 实现中的测试纪律（test-driven-development）

- **RED → GREEN → REFACTOR**：先写失败测试，看到失败，再写最小实现通过，再重构。
- 与「先写一大段再补测试」相反；技能中会强调反模式与纪律性。

### 6.6 任务间代码评审（requesting-code-review）

- 对照计划检查实现；按严重程度报告问题；**严重问题应阻塞继续往下做**，避免错误堆叠。

### 6.7 收尾（finishing-a-development-branch）

- 任务完成后：跑测试、整理选项（合并 / 提 PR / 保留 / 丢弃）、按需清理 worktree。

### 6.8 其他常用技能（按需触发）

| 技能 | 作用简述 |
|------|-----------|
| **using-superpowers** | 会话开始时的技能系统总览：何时必须加载技能、与平台工具名的对应等 |
| **systematic-debugging** | 系统化调试流程 |
| **verification-before-completion** | 完成/修复声明前的验证清单 |
| **dispatching-parallel-agents** | 多路独立任务并行分派 |
| **receiving-code-review** | 处理他人（或自己）评审意见时的严谨流程 |
| **writing-skills** | 编写或修改技能文档时的规范（贡献上游时常用） |

**总原则（官方强调）**：智能体在动手前应检查是否有适用技能；**有则必须跟**，不是「建议读一下」。

## 7. 其他

### 7.1 技能库一览（与 README 分类对齐）

下表为三列：**技能**（仓库 id）、**简介**（原则与能力边界）、**典型场景**（何时触发；细节以各 `SKILL.md` 为准）。

#### 测试

| 技能 | 简介 | 典型场景 |
|------|------|----------|
| `test-driven-development` | 先写失败测试再写实现，红—绿—重构；未看到失败则不能确信测对了。 | 新功能、修 Bug、重构、改行为前；一次性原型、生成代码、纯配置等例外需与人约定。 |

#### 调试

| 技能 | 简介 | 典型场景 |
|------|------|----------|
| `systematic-debugging` | 先根因、后补丁；反对症状式乱改。 | 测试失败、线上 Bug、异常行为、性能/构建/集成问题；赶时间、「一眼能修」、多轮修复无效时尤其要用。 |
| `verification-before-completion` | 声称完成/通过前必须先跑验证命令并读输出；证据先于断言。 | 提交前、开 PR 前、对外同步「做完了」之前。 |

#### 协作与流程

| 技能 | 简介 | 典型场景 |
|------|------|----------|
| `brainstorming` | 对话收敛为设计与规格；设计获批前不写实现。 | 新功能、组件、加能力、改行为等创造性工作前；极简需求也可缩成极短设计。 |
| `writing-plans` | 有规格且多步时，动代码前写可执行计划（小步、含路径与验证）。 | 设计已批、准备开工；常配 worktree；默认 `docs/superpowers/plans/...`。 |
| `executing-plans` | 加载计划、审阅后按任务执行，结束走收尾技能。 | 独立会话、带人类检查点执行书面计划；有子代理时上游更推荐 `subagent-driven-development`。 |
| `subagent-driven-development` | 每任务新子代理 + 两阶段评审（规格 → 质量）。 | 计划就绪、任务相对独立、本会话内快速迭代；Cursor 具备 Task/子代理时最贴上游。 |
| `dispatching-parallel-agents` | 多问题域并行分派，各带独立上下文。 | 多文件/子系统失败且根因独立、无共享可变状态；有顺序依赖或共享状态则不宜并行。 |
| `requesting-code-review` | 早评审、常评审；评审方拿精确上下文而非整段聊天。 | 子代理开发每任务后、大功能完成、合并主分支前；卡住或大规模重构前可选。 |
| `receiving-code-review` | 先读懂 → 用仓库事实核对 → 再决定是否改；禁止未验证就附和。 | 处理人评或 CI 评论；意见含糊或技术上可疑时先查证再改。 |
| `using-git-worktrees` | worktree 隔离目录与分支，带目录与安全校验。 | 与当前分支隔离开新功能、执行计划前要干净工作区、本地并行多条分支。 |
| `finishing-a-development-branch` | 测过再谈收尾：合并/PR/保留/丢弃与清理。 | 计划内任务全部完成且测试通过后。 |

#### 元

| 技能 | 简介 | 典型场景 |
|------|------|----------|
| `using-superpowers` | 技能怎么用、用户指令优先、多技能先后（流程类先于实现类）。 | 会话开始与每次动手前；有 1% 可能适用某技能就应先加载。 |
| `writing-skills` | 用 TDD 思路写/改 `SKILL.md`：压力场景 → 文档 → 验证智能体是否遵守。 | 自建或贡献技能、发布前验证；上游贡献须遵循该技能。 |
