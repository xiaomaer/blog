# AI 辅助开发流程：Skill（技能）

> 本文说明在 **Cursor** 等智能体环境中，**Skill（技能）** 是什么、为何有用、何时使用、如何编写，以及如何为前端团队沉淀技能。  
> 与 [Superpowers 流程](./AI%20辅助开发流程-superpowers.md) 的关系：Superpowers 把「做事方式」拆成多条可组合技能；本文聚焦 **Skill 这一载体本身** 与 **前端团队实践**。

---

## 1. 什么是 Skill

**Skill** 是一组写给智能体看的说明：通常以目录 + `SKILL.md` 的形式存在，在相关任务场景下被 **自动匹配** 或由你 **显式指定**，从而把「团队惯例、检查清单、输出格式、领域知识」注入到对话上下文中。

在 Cursor 中，常见形态包括：

| 要素 | 说明 |
| --- | --- |
| 存放位置 | **个人技能**：`~/.cursor/skills/<skill-name>/`；**项目技能**：`.cursor/skills/<skill-name>/`（可随仓库共享） |
| 核心文件 | `SKILL.md`（必填），含 YAML **frontmatter**（如 `name`、`description`）与正文指令 |
| 可选附件 | `reference.md`、`examples.md`、`scripts/` 等，用于减轻主文件体积、承载长文档或可执行脚本 |
| 与规则的关系 | **用户/项目规则**中的明确要求通常优先；Skill 在不冲突时补充「如何做某类任务」的细节 |

`description` 会参与「何时选用该技能」的判断：写清 **做什么** 与 **什么情况下该用**，智能体才容易在合适任务上自动加载对应技能。

---

## 2. 为什么需要 Skill

| 痛点 | Skill 能提供的价值 |
| --- | --- |
| 每轮对话重复贴「我们团队的 PR 模板、提交说明、目录约定」 | 一次写入 `SKILL.md`，需要时自动或按需加载 |
| 模型默认习惯与团队规范不一致（测试、命名、分层） | 把规范写成可执行的步骤与检查项，减少「接话就写」的偏差 |
| 新人或跨项目协作时上下文丢失 | 项目内 `.cursor/skills/` 随代码版本管理，与仓库同步演进 |
| 长对话挤占上下文 | 技能可被设计为「按需读取 reference」，比把所有规则塞进单条用户消息更可持续 |

简言之：**Skill 把可重复的「教模型怎么配合你」变成可版本化、可发现、可组合的资产**；与仅依赖系统提示或单次长 prompt 相比，更利于长期一致性与团队对齐。

---

## 3. Skill 适用场景

更适合用 Skill 承载的任务通常具备以下特征：

- **有稳定流程**：例如固定格式的 code review、发布前检查、依赖升级步骤。
- **有团队独有知识**：内部 API 约定、设计令牌路径、错误码表、与 Superpowers/OpenSpec 等流程配套的落盘目录。
- **希望智能体「看到任务就想起」**：通过 `description` 中的触发词（技术栈、文件类型、用户常用说法）提高自动命中率。
- **个人跨项目复用**：放在 `~/.cursor/skills/`，避免每个仓库复制粘贴同一套说明。

相对不必单独做一条 Skill 的情况：

- 一次性、不会再现的杂项；或几句话就能说清且无复用价值。
- 与现有规则完全重复、仅换表述的内容（应合并到规则或单一技能中，避免维护两份）。

## 4. 怎么创建 Skill

### 4.1 写之前先想清楚

建议先澄清（可写在技能开头或团队 wiki 里再给智能体引用）：

1. **目的与范围**：这条技能只解决哪一类问题？
2. **落盘位置**：个人全局 vs 当前仓库 `.cursor/skills/`。
3. **触发场景**：哪些用户说法、文件、目录应触发本技能？
4. **专有知识**：模型默认不知道但必须遵守的有哪些？
5. **输出形态**：是否需要固定模板（表格、章节、命令）？
6. **与现有资产的关系**：是否对齐 `AGENTS.md`、项目规则、已有脚本？

若用户提供了**必须逐字使用的原文**，应写入 `SKILL.md` 并保持原样，不要随意改写或扩写。

### 4.2 目录与 `SKILL.md` 骨架

```
skill-name/
├── SKILL.md              # 必填
├── reference.md          # 可选：长文档
├── examples.md           # 可选：示例
└── scripts/              # 可选：辅助脚本
```

`SKILL.md` 最小结构示例：

```markdown
---
name: your-skill-name
description: 用第三人称写清能力与触发条件（含关键词），便于自动选用。
---

# 技能标题

## 指令
分步骤、可执行的说明（假设模型已具备通用编程能力，只补充团队差异）。

## 示例
简短、可复制的对话或输入输出样例。
```

**注意**：不要将自定义技能放在 `~/.cursor/skills-cursor/`，该目录由 Cursor 内置技能占用。

### 4.3 写好 `description` 的技巧

- 使用**第三人称**（描述会进入系统侧上下文）。
- 同时写 **WHAT**（做什么）与 **WHEN**（何时用），并包含用户可能用到的 **触发词**。
- 控制长度（如 Cursor 要求 `description` 非空且有上限），把细节放进正文或 `reference.md`。

### 4.4 正文编写原则（摘要）

- **简洁**：只写模型缺的那部分；默认知识不必重复教学。
- **可执行**：步骤、检查清单优于泛泛原则。
- **可维护**：与真实命令、路径、分支策略一致，变更时只改技能一处。

安装与发现（若使用 CLI 生态）可参考同系列文档中的命令说明；在 Cursor 内也可通过规则声明「本项目启用的技能目录与命名约定」。

## 5. 前端开发常用的 Skill

更完整的选型表、组合建议与示例提示词见：[前端开发常用 skills](./前端开发常用skills.md)。此处仅作索引式摘要：

| 方向 | 典型技能名（示例） | 常见用途 |
| --- | --- | --- |
| React 性能与写法 | `react-best-practices`、composition 类 | 组件与数据流重构、避免常见性能坑 |
| Next.js 工程化 | `next-best-practices` | 路由、RSC 边界、数据获取与元数据 |
| 视觉与体验 | `frontend-design`、`ui-ux-pro-max` | 落地页/后台视觉与交互综合优化 |
| 规范与审查 | `web-design-guidelines` | 无障碍与界面规范评审 |
| 样式体系 | `tailwind-design-system` | 设计令牌与组件库样式统一 |

**使用建议**：按当前任务选 1～3 个技能即可，避免一次堆叠过多导致建议重叠或上下文浪费。

## 6. 创建前端开发团队的 Skill

团队级技能建议优先放在 **仓库** `.cursor/skills/`（或团队统一模板仓库），与 CI、目录结构、文档约定一起演进。可按三类主题拆分或合并为少数几条「粗粒度」技能，避免碎片化。

### 6.a 重复性工作流

适合封装为 Skill 的例子：

- **MR/PR 描述模板**：必须包含的章节（动机、方案、风险、测试说明）、链接到 issue 的格式。
- **依赖与锁文件策略**：何时用 `pnpm`/`npm`、是否允许 major 自动升级、安全补丁流程。
- **国际化/可访问性抽检步骤**：合并前在哪些页面跑哪些检查（可与脚本路径写进技能）。

编写要点：把「每次口头交代」改成 **编号步骤 + 完成定义（Definition of Done）**，必要时引用 `scripts/` 中的命令，避免在技能里写死易变的环境细节（可由规则覆盖）。

### 6.b 跨项目规范

适合沉淀为「可被多个仓库复制或子模块引用」的内容：

- **目录与命名**：features vs pages、`components/ui` 与业务组件边界。
- **API 与类型**：OpenAPI 生成客户端、错误对象形状、React Query key 约定。
- **设计与代码协作**：Figma token 与 Tailwind 主题文件对应关系、Storybook 发布约定。

编写要点：**单一事实来源**——技能正文只写摘要与必读规则，长表放 `reference.md`，并注明「以仓库内某路径文件为准」以免与实现漂移。

### 6.c 领域最佳实践

与「流程」不同，更偏 **判断标准与取舍**：

- 何时用服务端组件 vs 客户端组件；何时接受 bundle 增大换取 DX。
- 表单与复杂列表的性能与可维护性权衡清单。
- 与业务域相关的无障碍、国际化、合规用语（如金融、医疗前端展示约束）。

编写要点：用 **场景 → 推荐做法 → 反例** 结构，便于模型在模糊需求下对齐团队偏好；与 Superpowers 的「先规格后实现」结合时，可在技能中写明「本仓库是否强制 TDD / 设计落盘路径」。

---

## 小结

- **Skill** = 可版本化的智能体指令包，以 `SKILL.md` 为核心，靠 `description` 做发现、靠正文做执行。
- **团队**侧优先把重复流程、跨项目规范、领域最佳实践写成少量高质技能，并与项目规则、上游方法论（如 Superpowers）分工明确、避免冲突。
- **前端**选型可对照《前端开发常用 skills》，按任务组合使用，控制数量。
